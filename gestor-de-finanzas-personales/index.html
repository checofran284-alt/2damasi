"GESTOR DE FINANZAS PERSONALES"

"PROCESO DE RESOLUCIÓN"

 

En este documento se presenta el proceso de desarrollo de una aplicación web para la gestión de finanzas personales, cuyo objetivo principal es permitir al usuario registrar, organizar y consultar sus ingresos y gastos mediante una plataforma centralizada y accesible, además otras mejoras que se observaran durante la documentación. La aplicación busca facilitar el control económico personal, ofreciendo una visión clara de la situación financiera del usuario a través de una interfaz web intuitiva y multiplataforma.

El objetivo es proporcionar una experiencia de uso sencilla, organizada y visualmente clara, que permita interpretar rápidamente la información financiera mediante listados, códigos de color, tarjetas informativas y gráficas. De este modo, se evita la necesidad de utilizar métodos manuales como hojas de cálculo o aplicaciones poco flexibles, centralizando toda la información económica en un único sistema.

A lo largo de la práctica se describe cómo se ha diseñado la interfaz de la aplicación, la organización de las distintas pantallas y la gestión de los datos financieros. Para ello, se han utilizado componentes visuales que permiten identificar fácilmente los ingresos y gastos, la representación gráfica de estadísticas como el balance mensual o la distribución por categorías, y la incorporación de formularios interactivos que facilitan la creación, edición y eliminación de registros financieros.

Asimismo, se explica la implementación de una arquitectura cliente-servidor que permite almacenar y sincronizar la información del usuario en la nube mediante una base de datos, garantizando el acceso desde distintos dispositivos. El frontend se encarga de la interacción con el usuario, mientras que el backend gestiona la autenticación, las operaciones CRUD y el almacenamiento de datos, actuando como intermediario seguro entre la aplicación y la base de datos.

Durante el desarrollo se han tenido en cuenta aspectos como la organización del código, la separación entre frontend y backend, el almacenamiento local de datos mediante LocalStorage o IndexedDB para mejorar la experiencia sin conexión, y la gestión de ficheros asociados a los movimientos financieros. También se han considerado aspectos básicos de seguridad, asegurando que cada usuario solo pueda acceder a su propia información mediante un sistema de autenticación integrado.

Finalmente, se detallan las decisiones técnicas adoptadas y se proponen posibles mejoras futuras, como la incorporación de presupuestos automáticos, alertas ante gastos elevados, análisis financieros más avanzados, sincronización con entidades bancarias o la ampliación de funcionalidades en la aplicación móvil mediante el uso del APK generado para dispositivos Android.

Todo este proceso lo documentaremos mediante una serie de pasos los cuales se enfocarán en una guía de imágenes paso a paso...⬇️

Para este proceso de creación de mi prompt he tenido que recurrir al perplexity el cual después de muchas preguntas me dio esto:

Etapa 1➡️

Quiero que generes la estructura inicial de una aplicación web de finanzas personales llamada FINTRACK.
Tecnologías: HTML, CSS y JavaScript puro (sin frameworks). Código limpio y comentado.

Objetivo: luego empaquetaré esta web como APK Android, así que todo debe ser responsive y multipantalla.

Requisitos de diseño y funcionalidad para esta primera versión:

  1. ESTRUCTURA GENERAL

  • Una sola página index.html con layout tipo “app móvil”:

    • Barra superior con el nombre FINTRACK y botón de usuario.

    • Menú inferior tipo tabs con 4 secciones: “Gastos”, “Ahorro”, “Suscripciones”, “Inversiones”.

  • Cada sección será un panel distinto que se muestra/oculta con JavaScript al pulsar el tab.

  1. PANTALLA GASTOS (la más importante en esta versión)

  • Parte superior: resumen rápido del día/mes (total gastado, saldo aproximado).

  • Formulario para añadir un gasto:

    • Campos: importe, categoría (select: Comida, Transporte, Ocio, Otros), fecha, nota opcional.

    • Botón “Guardar gasto”.

  • Lista de gastos recientes debajo, en tarjetas.

  • Implementa en JavaScript:

    • Array en memoria para los gastos.

    • Guardar todos los gastos en localStorage (clave fintrack_gastos).

    • Al cargar la página, leer de localStorage y pintar la lista.

    • Función para calcular y mostrar el total del día/mes.

  1. BASE PARA EL RESTO DE SECCIONES (Ahorro, Suscripciones, Inversiones)

  • Crea solo la estructura visual y placeholders:

    • Un pequeño título y texto explicando qué irá en cada sección.

    • Un botón “Añadir nuevo…” que de momento solo muestra un alert o console.log.

  • Deja marcado con comentarios TODO en el JavaScript donde luego conectaremos con Supabase/Firebase y APIs externas.

  1. RESPONSIVE / MULTIPANTALLA

  • Usa CSS flexbox o grid para que:

    • Se vea bien en móvil (ancho 430px) y en escritorio.

    • En escritorio, el menú inferior puede estar en el lateral izquierdo.

  • Añade media queries básicas para cambiar el layout según resolución.

  1. CÓDIGO LIMPIO Y EXTENSIBLE

  • Escribe todo el JS en un archivo app.js separado.

  • Crea funciones claras: initApp(), showSection(id), addGasto(), saveGastosToLocalStorage(), loadGastosFromLocalStorage(), etc.

  • NO uses librerías externas.

  • Incluye comentarios que expliquen dónde conectaremos después:

    • Autenticación multiusuario (Supabase/Firebase).

    • CRUD en la nube.

    • Subida de ficheros (tickets, contratos).

    • Multiidioma.

Entrega en el Canvas:

  • index.html

  • styles.css

  • app.js
    con todo lo necesario para que la sección de Gastos ya funcione en local con localStorage.

Etapa 2➡️

Eres un especialista en diseño y desarrollo de aplicaciones de gestión de finanzas personales.
Eres también un desarrollador web senior experto en HTML, CSS y JavaScript puro, acostumbrado a crear interfaces responsive tipo app móvil y a dejar el código preparado para integrarse con backends (Supabase/Firebase, APIs externas, etc.).

Quiero que generes la estructura inicial de una aplicación web de finanzas personales llamada FINTRACK.

Tecnologías: HTML, CSS y JavaScript sin frameworks. El código debe ser claro, modular y bien comentado para que un estudiante de ciclo superior pueda entenderlo y ampliarlo.

Objetivo general

Esta web se empaquetará más adelante como APK Android. Debe funcionar bien en móvil y escritorio, y servir como base para añadir login, base de datos en la nube y más módulos.

Requisitos de diseño y funcionalidad (primera versión):

  1. Estructura general

  • Una sola página index.html con layout tipo app:

    • Barra superior con el nombre FINTRACK y un icono/botón de usuario (sin funcionalidad todavía).

    • Menú de navegación con 4 secciones: Gastos, Ahorro, Suscripciones, Inversiones.

  • Cada sección será un panel distinto que se muestra/oculta con JavaScript al pulsar el botón correspondiente.

  1. Pantalla GASTOS (ya funcional)

  • Encabezado con resumen rápido de:

    • Total gastado HOY.

    • Total gastado ESTE MES.

  • Formulario para añadir un gasto:

    • Importe (número).

    • Categoría (select: Comida, Transporte, Ocio, Suscripciones, Otros).

    • Fecha (por defecto, hoy).

    • Nota opcional.

    • Botón “Guardar gasto”.

  • Lista de gastos recientes en tarjetas, mostrando importe, categoría, fecha y nota.

  • Lógica en app.js:

    • Estructura de datos: array de objetos gastos.

    • Guardar y leer gastos de localStorage con una clave tipo "fintrack_gastos".

    • Al cargar la página:

      • Leer de localStorage.

      • Pintar la lista.

      • Calcular totales diario y mensual y mostrarlos en la UI.

    • Funciones claras: initApp(), addGasto(), renderGastos(), saveGastosToLocalStorage(), loadGastosFromLocalStorage(), calcularTotalesGastos().

    • Incluye validaciones básicas (no permitir importe vacío o cero).

  1. Pantallas Ahorro, Suscripciones e Inversiones (solo estructura y placeholders)

  • Cada sección debe tener:

    • Un título.

    • Un pequeño texto explicando qué se gestionará ahí.

    • Un botón “Añadir nuevo …” que, por ahora, solo haga console.log o muestre un alert.

  • En el JavaScript, deja funciones vacías o con TODO, por ejemplo:

    • addObjetivoAhorro(), addSuscripcion(), addInversion().

    • Comentarios claros indicando que más adelante se conectarán con una base de datos en la nube (Supabase/Firebase) y con APIs externas para precios de bolsa/criptos.

  1. Responsive / multipantalla

  • Usa CSS (flexbox o grid) para:

    • Vista móvil (ancho ~430px): menú de navegación en la parte inferior tipo barra de tabs.

    • Vista escritorio: menú en lateral izquierdo y contenido principal a la derecha.

  • Implementa media queries básicas para cambiar entre layout móvil y escritorio.

  • La app debe verse bien sin scroll horizontal en ambos casos.

  1. Organización del código

  • index.html: estructura semántica clara (header, main, nav, sections).

  • styles.css: estilos organizados por secciones, con comentarios.

  • app.js: todo el comportamiento:

    • Inicialización de la app.

    • Gestión de la navegación entre secciones.

    • Lógica de gastos y localStorage.

  • Añade comentarios donde se integrarán más adelante:

    • Autenticación de usuarios (multiusuario).

    • CRUD con Supabase/Firebase.

    • Subida y gestión de ficheros (imágenes de tickets, PDFs de contratos).

    • Multiidioma (por ejemplo, un objeto translations listo para usar).

Entrega en Canvas:

  • Un index.html, styles.css y app.js que permitan:

    • Navegar entre las 4 secciones.

    • Usar la sección de Gastos para añadir, guardar en localStorage, recargar la página y ver los gastos persistidos y los totales.

< Teniendo en cuenta que el prompt era muy largo decidí optar por partir el prompt a cachos para que el proyecto se pueda distribuir mejor con la IA y la propia no se sobrecargue.>

Etapa 3➡️

Me encuentro haciendo una práctica de DI de Desarrollo Web.

Quiero crear un dashboard de control de servidores con:

  • Backend en Node.js + Express,

  • Base de datos y autenticación con Supabase,

  • Frontend en HTML, CSS y JavaScript vanilla (sin React).

La app debe permitir:

  • Registro/login de usuario.

  • Ver una “ciudad dinámica” donde cada servidor es un poste sobre un mapa.

  • Al hacer clic en un poste, ver su estado (online, detenido, error), estadísticas (CPU/RAM simuladas) y botones para iniciar/parar/reiniciar.

  • Registrar cada acción en una tabla de logs.

Además, necesito que el backend ahora simule la ejecución de comandos, pero que quede claro en el código dónde podría ir, en el futuro, una llamada real por SSH a una máquina virtual.

Dame:

  1. El modelo de datos en Supabase (tablas servers, command_logs, y si lo ves útil server_stats) con sus campos.

  2. La estructura de archivos del proyecto (carpetas y ficheros para backend y frontend).

  3. Un resumen de la arquitectura (cómo se conectan frontend, Node y Supabase) explicado como para un estudiante de FP.

"CONCLUSIÓN"

Con la versión final de la aplicación de gestión de finanzas personales se logra una solución web completa, moderna y funcional que permite al usuario controlar de forma centralizada sus ingresos y gastos, consultar estadísticas financieras relevantes y gestionar su información económica desde una única interfaz. El proyecto destaca por su enfoque práctico, combinando un diseño visual cuidado —uso de códigos de color, tarjetas informativas, gráficas, modo claro/oscuro y una distribución clara de la información— con una estructura pensada para facilitar la comprensión rápida del estado financiero del usuario.

La aplicación se apoya en una arquitectura cliente-servidor bien definida, en la que el frontend se encarga de la visualización de datos y la interacción con el usuario, mientras que el backend gestiona la lógica de negocio, la autenticación y las operaciones CRUD sobre la base de datos en la nube. Esta separación garantiza la seguridad de la información, evitando accesos no autorizados y permitiendo una gestión eficiente y escalable de los datos financieros.

Además, la combinación de almacenamiento local y almacenamiento en la nube permite mejorar la experiencia de uso, facilitando el acceso a la información incluso en situaciones de conectividad limitada y asegurando la sincronización entre dispositivos. La integración de funcionalidades como el manejo de ficheros, el soporte multiidioma y la adaptación a diferentes resoluciones refuerzan el carácter multiplataforma de la aplicación, culminando en la generación de un APK funcional para dispositivos Android.

En conjunto, el resultado es una aplicación coherente y bien orientada al control financiero personal, accesible, segura y preparada para evolucionar. Como posibles mejoras futuras se plantean la incorporación de presupuestos automáticos, alertas inteligentes ante gastos excesivos, análisis financieros avanzados, integración con APIs bancarias y una mayor personalización de la experiencia del usuario.
