"MONITORIZACIÓN DE SERVIDORES"

"PROCESO DE RESOLUCIÓN"

En este documento se presenta el proceso de desarrollo de una aplicación web cuyo objetivo principal es la monitorización y control de servidores mediante un dashboard centralizado. La idea es crear una herramienta que permita al usuario visualizar de forma clara el estado de varios servidores, consultar estadísticas relevantes y ejecutar acciones remotas como iniciar, detener o reiniciar servicios, todo ello desde una interfaz web accesible desde el navegador.

El objetivo es ofrecer una experiencia de uso intuitiva, organizada y visualmente clara, facilitando la interpretación rápida del estado de los servidores mediante códigos de color, tarjetas informativas y gráficas, evitando la necesidad de acceder manualmente a cada servidor por separado mediante terminal.

A lo largo de la práctica se describe cómo se ha diseñado la interfaz del dashboard, cómo se gestionan los datos de los servidores y sus estados, y qué técnicas se han utilizado para mostrar la información de forma adecuada. Entre ellas se incluyen el uso de componentes visuales con indicadores de estado, la representación gráfica de estadísticas como el uso de recursos, y la incorporación de botones interactivos que

permiten ejecutar acciones sobre los servidores controlados.

Asimismo, se explica cómo se ha resuelto la limitación de los navegadores web para ejecutar comandos remotos directamente, implementando una arquitectura cliente-servidor en la que un backend actúa como intermediario entre la interfaz web y los servidores. Este backend se encarga de recibir las peticiones del usuario, ejecutar los comandos correspondientes en los servidores y devolver el resultado al frontend.

Durante el desarrollo también se han tenido en cuenta aspectos como la organización del código, la separación entre frontend y backend, la gestión de la configuración y el almacenamiento del estado de los servidores, así como consideraciones básicas de seguridad, evitando que el navegador tenga acceso directo a operaciones sensibles.

Finalmente, se detallan las decisiones técnicas adoptadas y se proponen posibles mejoras futuras, como la ampliación del número de servidores monitorizados, la incorporación de alertas automáticas ante incidencias, el almacenamiento histórico de métricas o la integración de sistemas de autenticación para el control de accesos.

Todo este proceso lo documentaremos mediante una serie de pasos los cuales se enfocarán en una guía de imágenes paso a paso...⬇️

Para este proceso de creación de mi prompt he tenido que recurrir al perplexity el cual después de muchas preguntas me dio esto:

Etapa 1➡️

Quiero que me ayudes a crear un dashboard web de control de servidores usando:

  • Backend en Node.js + Express.

  • Base de datos y autenticación en Supabase.

  • Frontend en HTML, CSS y JavaScript vanilla (sin React ni frameworks).

Objetivo funcional (versión 1 simulada)

  • Listar servidores con: nombre, host/IP (solo informativo), estado (online, stopped, error) y un color por estado.

  • Botones para iniciar, parar y reiniciar cada servidor.

  • Registrar cada acción en una tabla de logs (comando ejecutado, servidor, timestamp, resultado).

  • Estadísticas simples (CPU, RAM) simuladas.

Supabase

  1. Define el modelo de datos en Supabase:

    • Tabla servers con campos recomendados (id, name, host, status, etc.).

    • Tabla command_logs con campos (id, server_id, command, result, created_at).

    • Si lo ves útil, una tabla server_stats para CPU/RAM simuladas.

  2. Dame el esquema de tablas en texto (para copiar en el README) y ejemplos de queries SQL para crearlas.

Backend Node/Express
3. Diseña y genera código para estos endpoints:

  • GET /api/servers → devuelve la lista de servidores desde Supabase.

  • POST /api/servers/:id/action con body { action: "start" | "stop" | "restart" } →

    • Actualiza el estado del servidor en Supabase (simulación).

    • Inserta un registro en command_logs con el comando y un texto de resultado simulado.

  1. Usa el SDK oficial de Supabase para Node o llamadas HTTP si lo prefieres, con código claro y comentado.

SSH futuro (solo preparar el hueco)
5. Deja una función en el backend, por ejemplo async function executeRemoteCommand(server, action), que ahora solo simule la ejecución (devuelva un resultado de texto), pero comenta en el código dónde y cómo se podría usar una librería como ssh2 en Node para ejecutar comandos SSH reales en una máquina virtual.

Frontend
6. Proponme la estructura de archivos del frontend: index.html, styles.css, app.js.
7. Genera un ejemplo simple de:

  • index.html con una tabla de servidores y botones de acción.

  • app.js que llame a los endpoints del backend, pinte los colores según estado y actualice la vista sin recargar.

Formato de respuesta

  • Primero: resumen de la arquitectura.

  • Después: definición de tablas Supabase + SQL.

  • Luego: código del backend (Node/Express) con comentarios.

  • Por último: ejemplo básico de index.html y app.js.

  • Todo en español, explicado para un estudiante de FP que está aprendiendo.

Etapa 2➡️

Quiero crear un dashboard de control de servidores súper visual con Node + Supabase + HTML/CSS/JS vanilla.

Idea del frontend

  • Registro/Login con Supabase Auth (páginas separadas login.html y register.html).

  • Dashboard principal: mapa dinámico de ciudad (SVG o canvas simple) con postes de servidores como puntos interactivos.

  • Al clickear un poste: abre panel lateral/popup con detalles del servidor (nombre, estado/color, estadísticas CPU/RAM, botones start/stop/restart, logs recientes).

  • Animaciones: postes cambian color según estado y hacen efecto al interactuar.

Supabase

  1. Modelo de datos:

    • Tabla servers: id, name, host, status (online|stopped|error), color (#00ff00|#ff0000|#ffaa00), x, y (posición en mapa), user_id (FK).

    • Tabla command_logs: id, server_id, command, result, created_at.

    • Tabla server_stats (opcional): id, server_id, cpu, ram, updated_at.

  2. Esquema SQL y RLS básica para que cada usuario solo vea sus servidores.

Backend Node/Express
3. Endpoints:

  • GET /api/servers?userId=xxx → lista de servidores del usuario desde Supabase.

  • POST /api/servers/:id/action con { action: "start|stop|restart" } → simula comando, actualiza estado y guarda log.

  • Función executeRemoteCommand() comentada para SSH futuro con ssh2.

Frontend
4. Estructura archivos: login.html, register.html, index.html, styles.css, app.js, mapa.svg (ejemplo).
5. Código clave:

  • Login/register con Supabase Auth → guarda sesión localStorage y redirige.

  • index.html: fondo mapa SVG + JS que pinta postes desde API con posiciones x,y.

  • Click en poste → animación + panel lateral con datos y botones que llaman backend.

Genera

  • Esquema tablas Supabase.

  • Código backend Node (server.js con Express + Supabase client).

  • login.html y register.html básicos con Supabase Auth.

  • index.html + app.js que cargue servidores y pinte postes en SVG interactivos.

Explicación clara en español para estudiante FP. Al final, sugiéreme próximo prompt para pulir animaciones o añadir estadísticas.

Me encuentro haciendo una práctica de DI de Desarrollo Web.

La app debe permitir:

Además, necesito que el backend ahora simule la ejecución de comandos, pero que quede claro en el código dónde podría ir, en el futuro, una llamada real por SSH a una máquina virtual.

Dame:

  1. El modelo de datos en Supabase (tablas servers, command_logs, y si lo ves útil server_stats) con sus campos.

  2. La estructura de archivos del proyecto (carpetas y ficheros para backend y frontend).

  3. Un resumen de la arquitectura (cómo se conectan frontend, Node y Supabase) explicado como para un estudiante de FP.

"CONCLUSIÓN"

Con esta versión final del dashboard de monitorización y control de servidores se consigue una aplicación web completa y moderna que permite visualizar de forma centralizada el estado de distintos servidores, consultar estadísticas relevantes y ejecutar acciones remotas de gestión desde una única interfaz. El proyecto destaca por su enfoque práctico, combinando un diseño visual cuidado —uso de códigos de color, modo claro/oscuro, distribución clara de la información y transiciones suaves— con una estructura funcional orientada a facilitar la interpretación rápida del estado del sistema.

La aplicación se apoya en una arquitectura cliente-servidor bien definida, en la que el frontend se encarga de la presentación de la información y la interacción con el usuario, mientras que el backend gestiona la lógica de negocio y la ejecución de comandos en los servidores controlados. Esta separación permite mantener la seguridad del sistema, evitando la ejecución directa de operaciones sensibles desde el navegador, y facilita el mantenimiento y la escalabilidad de la aplicación.

Además, la utilización de mecanismos de almacenamiento y configuración adecuados permite gestionar el estado de los servidores y registrar las acciones realizadas, sentando las bases para una futura ampliación del sistema. El uso de máquinas virtuales como entorno de pruebas refuerza el carácter didáctico del proyecto, simulando un escenario real de administración de sistemas sin necesidad de infraestructura física adicional.

En conjunto, el resultado es una solución coherente y bien orientada a la monitorización y administración básica de servidores, accesible desde el navegador, multiplataforma y preparada para evolucionar. Entre las posibles mejoras futuras se encuentran la incorporación de alertas automáticas ante incidencias, el almacenamiento histórico de métricas, la gestión de usuarios y permisos, o la integración de nuevas visualizaciones que permitan un análisis más avanzado del estado del sistema.
